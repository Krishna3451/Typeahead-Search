# Performance Report

_Generated by `npm run benchmark` (in-process, no HTTP overhead) on the loaded dataset._

- Indexed queries: **300000** (of 591770 in the primary store)
- Prefix index build time: **4350.7 ms**
- Read workload: **20000** suggestion lookups (popularity-weighted prefixes)
- Write workload: **20000** search submissions

## 1. Suggestion latency (GET /suggest, popular mode)

Measured on the data path only (cache lookup -> on miss, prefix index + ranking -> cache fill).

| Metric | Value |
|---|---|
| p50 | 0.002 ms |
| p95 | 0.08 ms |
| p99 | 0.261 ms |
| max | 4.627 ms |
| mean | 0.017 ms |

**Cache hit rate (steady state): 80.92%** over 20000 measured lookups.
**DB reads on the suggestion path: 0** - suggestions are served entirely from the in-memory prefix index and the cache; the SQLite store is never read on a read request.

Trending mode (recency-aware re-rank, measured uncached): p95 0.397 ms.

## 2. Batch write reduction (POST /search)

Searches are aggregated in a write-behind buffer and flushed in batches, so repeated
queries collapse into a single additive UPSERT. Modelled on concentrated search traffic
(20000 submissions drawn from the top 1200 queries weighted by popularity),
flushed on a fixed cadence to emulate the server's periodic flush.

| | Naive (per-request) | Write-behind batching |
|---|---|---|
| Row writes | 20000 | 4714 |
| Transactions | 20000 | 5 |

- **Transaction reduction: 4000x** - one transaction per flush instead of one per
  search. This is the dominant win (far fewer fsyncs / round-trips) and holds regardless of skew.
- **Row-write reduction: 4.24x** - repeated queries collapse within a flush window;
  this scales with how concentrated the traffic is.
- Avoided: 15286 row writes and 19995 transactions.

**Failure trade-off:** a crash loses at most one un-flushed window. The append-only search
journal (`data/search-journal.log`) records every accepted search before buffering, so on
restart those un-flushed searches are replayed - bounding worst-case loss to searches not yet
fsync'd by the OS rather than a whole window.

## 3. Distributed cache - consistent hashing

4 logical cache nodes, 160 virtual nodes each. 10000 distinct query keys:

| Node | Keys | Share |
|---|---|---|
| cache-0 | 2619 | 26.2% |
| cache-1 | 2387 | 23.9% |
| cache-2 | 2488 | 24.9% |
| cache-3 | 2506 | 25.1% |

Ideal even share is 25%; virtual nodes keep every node within a few points of it.

**Elasticity:** adding a 5th node re-homed only **21%** of keys
(ideal ~20%). A plain `hash(key) % N` scheme would re-home roughly
75% of keys on the same change - that gap is the whole point of consistent hashing.
